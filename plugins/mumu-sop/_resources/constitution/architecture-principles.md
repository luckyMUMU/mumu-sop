# 架构原则

> **规范层级**: P0 级（宪法级规范）
> **约束强度**: 不可违背，违反即熔断

---

## 概述

本文档定义 SOP 流程中的**架构原则**，所有原则均为 P0 级，违反将触发熔断机制。

---

## 分层架构原则

### 原则描述

系统应按职责划分为清晰的层次，每层只依赖其下层。

### 标准分层

```
┌─────────────────────────────────────┐
│  表现层 (Presentation Layer)         │
│  - Controller / Handler             │
│  - DTO 转换                         │
│  - 请求验证                         │
└──────────────┬──────────────────────┘
               │ 只调用
               ▼
┌─────────────────────────────────────┐
│  应用层 (Application Layer)          │
│  - Service / UseCase                │
│  - 业务流程编排                      │
│  - 事务管理                         │
└──────────────┬──────────────────────┘
               │ 只调用
               ▼
┌─────────────────────────────────────┐
│  领域层 (Domain Layer)               │
│  - Entity / Aggregate               │
│  - Value Object                     │
│  - Domain Service                   │
│  - Domain Event                     │
└──────────────┬──────────────────────┘
               │ 只调用
               ▼
┌─────────────────────────────────────┐
│  基础设施层 (Infrastructure Layer)   │
│  - Repository 实现                   │
│  - 外部服务适配器                    │
│  - 数据库访问                       │
└─────────────────────────────────────┘
```

### 约束规则

| 规则 | 描述 | 验证方法 |
|------|------|----------|
| 单向依赖 | 上层依赖下层，下层不依赖上层 | ArchUnit |
| 跨层禁止 | 禁止跳过中间层直接调用 | ArchUnit |
| 接口隔离 | 层与层之间通过接口通信 | 代码审查 |

## 依赖方向原则

### 原则描述

依赖应指向稳定方向，高层模块不应依赖低层模块，两者都应依赖抽象。

### 依赖倒置

```
┌─────────────────┐     ┌─────────────────┐
│  高层模块        │     │  抽象接口        │
│  (Service)      │────▶│  (Interface)    │
└─────────────────┘     └────────┬────────┘
                                 │
                                 │ 实现
                                 ▼
                        ┌─────────────────┐
                        │  低层模块        │
                        │  (Repository)   │
                        └─────────────────┘
```

### 约束规则

| 规则 | 描述 | 验证方法 |
|------|------|----------|
| 依赖抽象 | 模块间通过接口依赖 | 代码审查 |
| 稳定依赖 | 依赖应指向稳定方向 | 依赖分析 |
| 无循环依赖 | 模块间禁止循环依赖 | dependency-cruiser |

---

## 模块边界原则

### 原则描述

模块应具有高内聚低耦合的特性，模块边界应清晰明确。

### 模块划分标准

| 标准 | 描述 |
|------|------|
| 业务边界 | 按业务领域划分（如订单、支付、库存） |
| 技术边界 | 按技术职责划分（如认证、日志、缓存） |
| 团队边界 | 按团队职责划分（如前端、后端、数据） |

### 模块通信规则

```
模块 A ←──→ 模块 B
   │          │
   │  通信方式 │
   │          │
   ▼          ▼
┌─────────────────┐
│ 事件驱动 / API  │
│ 消息队列 / RPC  │
└─────────────────┘
```

### 约束规则

| 规则 | 描述 | 验证方法 |
|------|------|----------|
| 单一职责 | 每个模块只有一个变化原因 | 代码审查 |
| 接口隔离 | 模块间通过定义良好的接口通信 | API 契约验证 |
| 依赖最小 | 模块间依赖应最小化 | 依赖分析 |

## 技术选型边界

### 原则描述

技术选型应遵循"少即是多"原则，优先使用项目已有技术栈。

### 技术选型流程

```
需求提出
    ↓
评估必要性（是否真的需要？）
    ↓
调研替代方案（至少 3 个候选）
    ↓
验证兼容性（与现有技术栈兼容？）
    ↓
记录决策理由（ADR）
    ↓
技术委员会审批
    ↓
引入并配置
```

### 约束规则

| 规则 | 描述 | 验证方法 |
|------|------|----------|
| 优先复用 | 优先使用项目已有技术 | 依赖审查 |
| 避免重复 | 禁止引入功能重复的库 | 功能对比 |
| ADR 记录 | 重要技术选型必须记录 ADR | 文档审查 |

### 禁止引入

- 功能重复的库或工具
- 为"可能的"需求提前引入
- 维护不活跃的库（>1 年无更新）
- 社区支持差的库

## DDD 战术模式应用

### 聚合根

| 规则 | 描述 |
|------|------|
| 唯一入口 | 聚合根是访问聚合内实体的唯一入口 |
| 边界保护 | 聚合外不能直接修改聚合内实体 |
| 事务边界 | 一个聚合 = 一个事务 |

### 值对象

| 规则 | 描述 |
|------|------|
| 不可变 | 值对象创建后不可修改 |
| 相等性 | 值对象通过属性值判断相等 |
| 自验证 | 值对象构造时验证自身有效性 |

### 领域服务

| 规则 | 描述 |
|------|------|
| 无状态 | 领域服务不应持有状态 |
| 跨实体 | 封装跨实体的业务逻辑 |
| 领域语言 | 方法名应使用领域语言 |

### 仓储模式

| 规则 | 描述 |
|------|------|
| 接口在领域层 | 仓储接口定义在领域层 |
| 实现在基础设施层 | 仓储实现在基础设施层 |
| 隐藏实现细节 | 不暴露数据库细节 |

## 架构决策记录（ADR）

### ADR 模板

```markdown
# ADR-XXX: [决策标题]

## 状态
[提议/已接受/已弃用]

## 背景
[决策背景描述]

## 决策
[决策内容]

## 理由
[决策理由]

## 后果
[决策带来的影响]

## 替代方案
[考虑过的其他方案]
```

### ADR 触发条件

- 涉及架构层面的技术选型变更
- 影响多个模块的接口变更
- 安全、性能、可用性等非功能性决策
- 与既有 ADR 存在冲突的决策

## 层级驱动原则

> **核心原则**: SOP 基于约束树执行，设计从根开始，实现从叶子开始

### 约束树核心结构

约束树以多叉树形态存在，定义了长效约束的层级关系：

```
┌─────────────────────────────────────────────────────────┐
│  P0 根节点（工程宪章）                                    │
│  - 安全、质量、架构红线                                   │
│  - 不可违背                                              │
└──────────────────────┬──────────────────────────────────┘
                       │ 派生
                       ▼
┌─────────────────────────────────────────────────────────┐
│  P1 一级子节点（系统规范）                                │
│  - 性能、可用性、接口约束                                 │
│  - 继承 P0 约束                                          │
└──────────────────────┬──────────────────────────────────┘
                       │ 派生
                       ▼
┌─────────────────────────────────────────────────────────┐
│  P2 二级子节点（模块规范）                                │
│  - 代码质量、文档、测试约束                               │
│  - 继承 P1 约束                                          │
└──────────────────────┬──────────────────────────────────┘
                       │ 派生
                       ▼
┌─────────────────────────────────────────────────────────┐
│  P3 三级子节点（实现规范）                                │
│  - 编码规范、注释、Git 规范                              │
│  - 继承 P2 约束                                          │
│  - 可挂载临时子节点（独立存储，引用关联）                  │
└─────────────────────────────────────────────────────────┘
```

### 约束从属关系

| 关系类型 | 描述 | 示例 |
|----------|------|------|
| 父节点 | 该约束继承自哪个约束 | P2 的父节点是 P1 |
| 子节点 | 该约束派生出哪些约束 | P1 的子节点是多个 P2 |
| 兄弟节点 | 同一父节点下的其他约束 | 同一 P1 下的多个 P2 |

### 约束继承规则

| 规则 | 描述 | 验证方法 |
|------|------|----------|
| 不违反父约束 | 子约束不得违反父约束的规定 | 约束检查 |
| 可细化要求 | 子约束可以细化父约束的要求 | 设计审查 |
| 可扩展范围 | 子约束可以扩展父约束的范围 | 设计审查 |
| 影响评估 | 父约束变更时必须评估对子约束的影响 | 变更审查 |

### 设计从根开始

设计必须从约束树根节点（P0）开始，按照层级顺序逐层向下：

```
设计流程（自顶向下）:

P0-Design (工程宪章设计)
    │
    │ 用户审查确认
    ▼
P1-Design (系统架构设计)
    │
    │ 用户审查确认
    ▼
P2-Design (模块设计)
    │
    │ 用户审查确认
    ▼
P3-Design (实现设计)
    │
    │ 用户审查确认
    ▼
临时子节点创建 (spec.md + tasks.md + checklist.md)
```

**设计层级依赖规则**：
- 必须先完成父节点的设计
- 设计必须参考父节点的约束
- 设计完成后必须进行护栏检查
- 用户确认后才能进入下一层级设计

### 实现从叶子开始

实现必须从约束树叶子节点开始，按照层级顺序逐层向上：

```
实现流程（自底向上）:

临时子节点实现 (如果存在)
    │
    │ 护栏检查
    ▼
P3-Implement (实现规范层)
    │
    │ 护栏检查
    ▼
P2-Implement (模块规范层)
    │
    │ 护栏检查
    ▼
P1-Implement (系统规范层)
    │
    │ 护栏检查
    ▼
P0-Implement (工程宪章层 - 约束验证)
```

**实现层级依赖规则**：
- 必须先完成子节点的实现
- 实现必须满足该节点的约束
- 实现完成后必须进行护栏检查
- 验证通过后才能进入上一层级实现

### 临时子节点机制

临时子节点独立存储，通过引用关联到 P3 节点：

```
临时子节点存储结构:

.trae/specs/{change-id}/
├── spec.md        # 任务规范文档，定义临时约束
├── tasks.md       # 任务列表，定义执行步骤
└── checklist.md   # 检查清单，定义验证标准

元数据:
  - 引用的 P3 节点 ID
  - 创建时间
  - 任务状态
```

**临时子节点生命周期**：
1. 创建：独立存储，引用 P3 节点
2. 执行：继承 P3 及其祖先节点的约束
3. 完成：归档，解除引用关系
