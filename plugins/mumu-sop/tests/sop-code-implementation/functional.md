# sop-code-implementation 功能测试

## 测试 1: TDD 红阶段 - 编写失败的测试

### 前置条件
- 设计文档已通过审查
- 规范文档已确认
- 技术栈已确定

### 输入
```
设计文档定义：
class OrderService {
  createOrder(items: OrderItem[]): Order;
  cancelOrder(orderId: string): Result<void, Error>;
}
```

### 预期行为
1. 根据设计文档识别功能点
2. 调用 sop-test-implementation 编写测试
3. 运行测试，验证测试失败（功能未实现）

### 验证点
- [ ] 是否先创建测试文件
- [ ] 测试是否因功能未实现而失败
- [ ] 未跳过红阶段直接实现

---

## 测试 2: TDD 绿阶段 - 最小实现

### 前置条件
- 红阶段测试已失败

### 输入
```
失败的测试：OrderService.createOrder 应创建订单
```

### 预期行为
1. 创建最小代码结构
2. 实现足以使测试通过的代码
3. 运行测试，验证测试通过
4. 不实现测试之外的额外功能

### 验证点
- [ ] 测试是否通过
- [ ] 代码是否是最小实现
- [ ] 是否无多余功能

### 前置条件
- 绿阶段测试已通过

### 验证点

## 测试 4: 完整 TDD 循环

### 输入
```
设计文档：
class Calculator {
  add(a: number, b: number): number;
  divide(a: number, b: number): Result<number, Error>;
}
```

### 预期行为（按顺序）
1. **红阶段**: 为 add 和 divide 编写失败测试
2. **绿阶段**: 最小实现使测试通过
3. **重构阶段**: 优化代码，测试仍通过
4. **质量检查**: lint、类型检查、覆盖率

### 验证点
- [ ] 是否遵循 红→绿→重构 顺序
- [ ] 是否每个阶段都有验证
- [ ] 最终代码是否通过所有检查
